# Product Context Model

## Purpose

Use this reference to create `PRODUCT.md` before `DESIGN.md`.

The design system must be a response to product context, not a decoration layer.

## Required fields

1. Product name
2. Product category
3. Core users
4. User situation
5. Primary job to be done
6. Product promise
7. Emotional register
8. Brand traits
9. Must never feel like
10. Main UX risks
11. Accessibility needs
12. Localization needs
13. Writing tone
14. Anti-references
15. Success criteria
16. Assumptions

## How to infer responsibly

If the user gives little context, infer carefully from the category.

Example:

A lead generation SaaS likely needs speed, credibility, control, and data clarity.

Do not infer playful mascots, crypto-like gradients, or theatrical animation unless the user asks.

## Product promise test

A product promise should answer:

"After using this, the user can do what with less pain, less risk, or more clarity?"

Bad:

"The smartest AI lead generation platform."

Better:

"A workspace that helps GTM operators collect, verify, and organize leads without turning the process into spreadsheet cleanup."
